## О проекте
Приложение для поиска блюд по названию и просмотра детальной информации о рецепте.

Стек проекта:
- Compose
- Navigation Compose
- ViewModel + StateFlow
- Coroutines + viewModelScope
- Retrofit
- Room
- Hilt

Используемое API: [TheMealDB](https://www.themealdb.com/api.php)

Основные эндпоинты:
- `GET /api/json/v1/1/search.php?s=` — поиск блюд по названию
- `GET /api/json/v1/1/lookup.php?i=` — получение полной информации о блюде по `id`

## Что сделано в ДЗ 4
- Реализовано 2 основных экрана: поиск и детали блюда.
- Подключена Navigation Compose с переходом на экран `Dish/{id}`.
- Реализованы UI-состояния `Loading / Error / Retry / Empty / Success`.
- Подключён Hilt для DI.
- Подключён Room для истории просмотров.
- При открытии блюда оно сохраняется в историю, история переживает перезапуск приложения.

Как проверить сценарий из ДЗ 4:
1. Запустить приложение.
2. Найти блюдо, например `chicken`.
3. Открыть любое блюдо.
4. Перейти на вкладку истории и убедиться, что блюдо сохранилось.
5. Перезапустить приложение и проверить, что история сохранилась.

## Что сделано в ДЗ 5

Сделано:
- `11` юнит-тестов
- `3` интеграционных теста

### Юнит-тесты
Покрытые сценарии:
- корректное начальное состояние `FoodViewModel`
- пустой поисковый запрос
- успешный поиск блюда
- ошибка при поиске
- загрузка деталей для несуществующего блюда
- `retry()` после ошибки с корректной новой попыткой загрузки
- устаревший результат поиска не перетирает более новый
- устаревший результат загрузки деталей не перетирает более новый
- корректное преобразование `RawMeal -> Dish`
- корректное преобразование `RawMeal -> DishDetails`
- корректная обработка пустого ответа API

### Интеграционные тесты
Покрытые сценарии:
- `HistoryStorage + Room`: повторная запись одного и того же блюда не создаёт дубль
- `UI + Navigation`: список -> клик по элементу -> переход на экран деталей нужного `id`
- `UI`: ошибка -> нажатие `Retry` -> успешное состояние

### Flow / StateFlow
Проверяются последовательности эмиссий:
- `Empty -> Loading -> Found`
- `Empty -> Loading -> Failure`
- `Loading -> Failure`
- `Loading -> Ready` после повторной попытки

Проверяемое нетривиальное потоковое поведение:
- `retry()` действительно инициирует новую попытку запроса
- состояние после ошибки и повторной загрузки меняется в корректной последовательности
- устаревший результат не попадает в UI после нового запроса
